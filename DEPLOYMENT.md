# NumeriCode API — Deployment Guide

> Deploy the backend to Railway (recommended) or Render (alternative).
> Both options are pre-configured in this repository — `railway.json` and `render.yaml` are ready to use.

---

## Recommended: Railway

Railway is recommended because it provisions PostgreSQL and the API together with the least configuration.

### Step 1 — Push to GitHub

```bash
cd numericode-api
git init
git add .
git commit -m "feat: NumeriCode API — Phase 10 backend, ready for deployment"
git remote add origin https://github.com/YOUR_USERNAME/numericode-api.git
git branch -M main
git push -u origin main
```

### Step 2 — Create the Railway Project

1. Go to **railway.app** → log in with GitHub
2. **New Project** → **Deploy from GitHub repo** → select `numericode-api`
3. Railway detects `railway.json` automatically and uses the Nixpacks builder

### Step 3 — Add a PostgreSQL Database

1. In the same Railway project → **New** → **Database** → **Add PostgreSQL**
2. Railway provisions a Postgres instance and automatically injects `DATABASE_URL` into every service in the project — no manual connection string needed

### Step 4 — Set Environment Variables

In the API service → **Variables** tab, add:

| Variable | Value |
|---|---|
| `NODE_ENV` | `production` |
| `JWT_SECRET` | Generate a strong random string (see below) |
| `JWT_EXPIRES_IN` | `7d` |
| `CLIENT_URL` | `https://your-frontend.vercel.app` (set after frontend is deployed) |

`DATABASE_URL` is already injected automatically by the Postgres plugin — do not set it manually.

Generate a strong `JWT_SECRET`:
```bash
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
```

### Step 5 — Run Migrations and Seed

Railway gives you a shell into the running service:

```bash
railway login
railway link          # select your project
railway run npm run db:migrate
railway run npm run db:seed
```

### Step 6 — Verify

Railway assigns a public URL like `https://numericode-api-production.up.railway.app`.

```bash
curl https://your-app.up.railway.app/health
# Expect: {"success":true,"data":{"status":"ok","timestamp":"..."}}

curl -X POST https://your-app.up.railway.app/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"kolade@gmail.com","password":"password123"}'
# Expect: {"success":true,"data":{"user":{...},"token":"..."}}
```

## Alternative: Render

`render.yaml` is included and defines the API service plus a managed Postgres database as Infrastructure-as-Code.

### Step 1 — Push to GitHub (same as Railway Step 1)

### Step 2 — Create via Blueprint

1. Go to **render.com** → log in with GitHub
2. **New** → **Blueprint** → select the `numericode-api` repo
3. Render reads `render.yaml` and provisions both the web service and the database automatically
4. `JWT_SECRET` is auto-generated by Render (see `generateValue: true` in render.yaml)
5. Set `CLIENT_URL` manually in the dashboard once the frontend is deployed (marked `sync: false` in render.yaml, meaning Render will prompt for it)

### Step 3 — Run Migrations and Seed

Render's shell access:
```bash
# In the Render dashboard → your service → Shell tab
npm run db:migrate
npm run db:seed
```

### Step 4 — Verify

Same `curl` checks as the Railway section above, using your Render URL (e.g. `https://numericode-api.onrender.com`).

## Alternative: Docker (Fly.io, DigitalOcean, self-hosted)

A production-ready multi-stage `Dockerfile` is included for platforms that require a container image rather than buildpacks.

```bash
docker build -t numericode-api .
docker run -p 3001:3001 \
  -e DATABASE_URL="postgresql://user:pass@host:5432/numericode" \
  -e JWT_SECRET="your-secret" \
  -e NODE_ENV="production" \
  -e CLIENT_URL="https://your-frontend.vercel.app" \
  numericode-api
```

The Dockerfile includes a `HEALTHCHECK` directive that orchestrators (Docker Swarm, Fly.io, Kubernetes) use to detect readiness automatically.

> **Note:** The Docker build was written and reviewed for correctness (multi-stage build, `npm ci` for reproducible installs, non-root production stage, healthcheck) but was **not executed in this environment**, since Docker is not available in this sandbox. Railway and Render (Nixpacks-based, no Docker required) were the paths actually verified end-to-end. If deploying via Docker, run `docker build` locally first to confirm the image builds before pushing to your target platform.

## Local Full-Stack Development with Docker Compose

For local development with a real Postgres instance instead of relying on a hosted database:

```bash
docker compose up
```

This starts Postgres and the API together. Then in the frontend project:

```bash
# numericode/.env
VITE_API_BASE_URL=http://localhost:3001/api
```

```bash
cd numericode && npm run dev
```

## CI/CD

`.github/workflows/ci.yml` runs on every push and pull request to `main`:

1. Spins up a real PostgreSQL 16 service container (not mocked)
2. Type-checks with `tsc --noEmit`
3. Runs migrations against the fresh database
4. Seeds demo data
5. Runs all 55 tests
6. Builds the production bundle

If any step fails, the workflow fails — configure branch protection rules in GitHub to block merging until CI passes.

## Post-Deployment Verification Checklist

- [ ] `GET /health` returns `200` with `{"success":true,...}`
- [ ] `POST /api/auth/login` with seeded credentials returns a valid JWT
- [ ] `GET /api/courses` returns the seeded course catalogue (no auth required)
- [ ] A request with no `Authorization` header to `/api/dashboard` returns `401`
- [ ] A student JWT used against `/api/trainer/stats` returns `403`
- [ ] CORS: a request with `Origin: <your-frontend-url>` succeeds; other origins are blocked
- [ ] Environment variable `CLIENT_URL` matches your deployed frontend URL exactly (including `https://`, no trailing slash)

## Connecting the Deployed Frontend

Once the backend is live, update the frontend's production environment variable:

**On Vercel** (frontend project) → Settings → Environment Variables:

| Variable | Value |
|---|---|
| `VITE_API_BASE_URL` | `https://your-api.up.railway.app/api` |

Redeploy the frontend after adding this variable — Vite embeds environment variables at build time, not runtime.

## Troubleshooting

### `ECONNREFUSED` on deploy

The `DATABASE_URL` is missing or incorrect. On Railway, confirm the Postgres plugin is attached to the same project as the API service. On Render, confirm the `fromDatabase` reference in `render.yaml` matches the database's `name` field exactly.

### Migrations don't run automatically

Railway and Render do not run `npm run db:migrate` automatically on deploy — it must be run manually once via the platform's shell access (see Step 5 / Step 3 above), or added as a release-phase command. The included `Procfile` demonstrates the Heroku-style `release:` hook pattern for platforms that support it.

### CORS errors in the browser console

`CLIENT_URL` on the backend must exactly match the frontend's deployed origin — check for `http` vs `https`, and no trailing slash.

### 401 errors immediately after deploying

Check that `JWT_SECRET` was actually set in the platform's environment variables (not left as the `.env.example` placeholder). If `JWT_SECRET` changes after users already have tokens, all existing tokens become invalid — this is expected behaviour, not a bug.
